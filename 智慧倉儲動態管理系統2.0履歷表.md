## 🌟 亮點精華版 (適合放在 104 專案經驗簡介 / 限制字數區塊)

**【專案名稱】** 智慧倉儲動態管理系統 2.0 (獨立全端開發)
**【技術棧】** React, Vite, TailwindCSS, Node.js, Express, SQLite, Axios, framer-motion, html5-qrcode
**【專案簡介】**
為解決傳統倉儲找料困難、庫存盤點與異動效率低落等痛點，從零到一獨立設計並開發之現代化 WMS 系統。捨棄厚重的套裝軟體，採用前後端分離架構與輕量級資料庫，實現跨平台即時庫存追蹤與可視化管理。
**【核心貢獻與亮點】**
1. **首創 2D 動態儲位地圖**：不依賴第三方 Canvas 套件，純手工利用 React 狀態結合 CSS Grid 動態演算法，將冰冷的料架資料轉化為可互動的 2D 視覺地圖，支援動態閃爍指引，找料時間縮短 50% 以上。
2. **行動端 PDA 掃描化**：整合 WebRTC 相機 API (`html5-qrcode`)，克服 iOS/Android 瀏覽器 HTTPS 安全限制，將一般智慧型手機直接轉化為出入庫掃描設備。內建防呆鎖定機制，大幅降低 90% 人工輸入錯誤率。
3. **安全庫存與效能最佳化**：後端實作 SQLite Transaction 批次處理，支援萬筆 Excel 資料秒級匯入覆蓋。搭載安全庫存預警引擎，自動分析低水位料件並產出「建議購料 Excel 清單」，實現採購自動化。

## 🚀 深度技術剖析版 (適合面試官深挖 / 個人亮點頁面 / GitHub Readme)

如果你需要跟技術主管面試，可以準備好這套說詞：

### 1. 突破性的 UI/UX：二維網格地圖渲染演算法
- **痛點**：舊系統的儲位只有純文字代碼（如 4A-05-1），新人常找不到料在哪裡。
- **技術解法**：我利用 `useMemo` 與 `CSS Grid` 屬性，在前端寫了一套自動排版演算法。它會根據後端的資料庫 X, Y 座標，動態撐開一個虛擬格子。遇到走道或大型障礙物（跨越多個網格的物件），我透過解析特定的後端自訂標籤（如 `#V_#`）來設定 `grid-column: span N`，並精準剔除被覆蓋的底層節點 (Covered Nodes) 避免 DOM 重疊效能災難。
- **亮點**：這套系統能讓倉管看著畫面就知道要走到第幾條走道，結合 `framer-motion` 在 BOM 表出庫時給予紅色 Ping 動畫指引，體驗極佳。

### 2. 解決行動裝置跨平台硬體調用難題
- **痛點**：採購專用的工業 PDA 掃描槍太貴，希望員工直接用手機瀏覽器掃描。
- **技術解法**：為了解決現代手機瀏覽器對於相機權限的嚴格限制（Secure Context），我透過設定 Vite 的 `@vitejs/plugin-basic-ssl` 憑證使本地端支援 HTTPS 連線。在前端串接原生 `Html5Qrcode` 時，遭遇了 React `useEffect` 非同步渲染與相機鏡頭搶佔 DOM 的 Race Condition (競爭危害) 當機問題。最終透過 `setTimeout` 延遲掛載與閉包 (Closure) 狀態解耦成功修復。
- **亮點**：我還實作了 UX 防呆機制——出庫掃描時如果偵測到該儲位為空，會立即跳出紅色警告攔截操作，避免資料庫產生負庫存髒資料。

### 3. 資料庫的輕量化與資料一致性 (Data Consistency)
- **痛點**：工廠有幾萬筆初期的庫存與 Excel 盤點表需匯入，普通的逐筆 API 呼叫會導致整個伺服器卡頓數十秒。
- **技術解法**：後端刻意選用 `better-sqlite3`（同步執行、超低延遲特性）取代肥大的 MySQL，並在 Excel 解析完畢後，透過 `db.transaction()` 包裝所有的 `INSERT / UPDATE / DELETE` 操作。不僅保證了萬筆資料匯入只需幾百毫秒，更達成了 ACID 特性——一旦中間有欄位出錯，所有資料會徹底 Rollback 放棄寫入，絕不產生孤兒資料 (Orphan Data)。

## 🎯 面試 Q&A 猜題小抄 (你可以這樣引導面試官)

* **Q: 為什麼不用關聯式大廠資料庫如 MySQL/PostgreSQL？**
  * **A**: 考量到這個專案初期是為了能「隨插即用」、一鍵啟動 (透過我寫好的 `.bat` 自動尋找 Node 路徑)。SQLite 的檔案型特性讓部署成本降到最低，搭配 `better-sqlite3` 的同步驅動以及我的 Transaction 在幾萬筆庫存的規模下，效能甚至超越需要走 TCP/IP 網路通訊的 MySQL。
* **Q: 前端地圖資料那麼多，React 畫面不會卡頓嗎？**
  * **A**: 確實遇到過！為此我將地圖矩陣的計算邏輯抽離，並使用 `useMemo` 給包裝起來。只有當收到新的地圖 API Payload 時才會重新走訪 X, Y 迴圈計算 DOM 結構，避免了元件頻繁 re-render 帶來的效能瓶頸。
* **Q: 專案中遇到最大的 Debug 挑戰是什麼？**
  * **A**: 是手機相機開發！因為我在桌機測試沒問題，一上手機就因為 HTTP 限制被擋，解決了憑證後，又因為 React 重新整理組件太快，舊的相機實例還沒關閉就被迫重啟導致白畫面。這讓我深刻體會到 React 生命週期 (Component Lifecycle) 與外部第三方原生 JS 實例共存時，需要特別處理清除函式 (`cleanup function`) 的重要性。
