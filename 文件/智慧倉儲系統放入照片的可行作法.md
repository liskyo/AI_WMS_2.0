# 智慧倉儲系統：料件圖片功能導入可行性分析與最佳實務作法

本文件旨在評估於現有「智慧倉儲動態管理系統 2.0 (Smart WMS)」中，未來導入「料件圖片 (Item Images)」功能的最佳儲存策略與系統影響分析。

---

## 摘要與核心結論

**結論：強烈建議採用「檔案系統儲存圖片實體＋SQLite 儲存相對路徑」的混合式架構。**

不建議將圖片以 BLOB (Binary Large Object) 格式直接存入現有的 `better-sqlite3` 資料庫中。雖然技術上可行，但會對現有極低延遲的讀寫效能與免安裝部署的輕量化優勢造成破壞性影響。

---

## 一、方案評估與深度比較

### 方案 A：完全存入 SQLite 資料庫 (BLOB)
*   **做法**：在 `items` 料件主檔的資料表中新增一個 BLOB 欄位，將圖片的二進位編碼直接塞進去。
*   **❌ 缺點**：
    1.  **檔案暴增**：若有 5,000 筆料件圖片（每張 1MB），`warehouse.db` 檔案體積將從幾 MB 直接膨脹到 5GB。
    2.  **效能災難**：檔案型資料庫在處理大型檔案時，讀寫鎖定 (I/O Lock) 時間會拉長。原本 WMS 最自豪的「秒級萬筆覆蓋匯入/回滾」將面臨卡頓風險。
    3.  **快取困難**：每次前端請求圖片，後端伺服器都要做一次資料庫查詢，並將 BLOB 轉回 Base64 或 Image Stream，這是極為浪費 CPU 效能的行為，且瀏覽器難以進行原生快取 (HTTP Cache)。

### 方案 B：檔案系統 + 路徑儲存 (業界標準 / 最佳實踐 ✅)
*   **做法**：
    *   **實體照片**：存放在 Node.js 後端伺服器的一個資料夾內（如 `server/public/item_images/A001.jpg`）。
    *   **SQLite 資料庫**：在 `items` 表新增 `image_url` 欄位（儲存字串類型如 `"/images/A001.jpg"`）。
    *   **前端取用**：遇到需要顯示圖片時，React 直接使用 `<img src={item.image_url} />` 向 Express 伺服器請求靜態資源。
*   **✅ 優點**：
    1.  **資料庫極致純淨**：`warehouse.db` 依然只存純文字，體積與效能不會受到任何影響。
    2.  **後端靜態派發超快**：Express.js 會直接把圖片當作靜態資源 (Static Assets) 丟給前端，不會佔用資料庫 Event Loop 與 I/O 資源。
    3.  **前端瀏覽器原生快取得利**：第一次載入圖片後，瀏覽器會自動把 URL Cache 起來，後續作業員高頻切換畫面時，圖片幾乎是「瞬間出現 (0 毫秒延遲)」。
    4.  **相容未來雲端擴展**：若未來圖片無限增長，只要把 `image_url` 的字串改成指向 Amazon S3 或是企業內部 NAS 連結即可，程式邏輯不須大改。

## 二、導入最佳實務方案 (方案 B) 的實作藍圖

若未來決定開展此功能，可依循以下開發 Roadmap，這不會影響現有穩定運行的架構：

### 階段一：資料庫微整形 (Database Migration)
*   在 `server.js` 的 `initDb` 中增加非破壞性的 Migration 語法：
    ```sql
    ALTER TABLE items ADD COLUMN image_url TEXT;
    ```
*   這項操作只需 0.1 秒，不會動到現有任何庫存、地圖或權限邏輯。

### 階段二：後端伺服器擴展 (Backend Expansion)
1.  **靜態資料夾掛載**：利用 Express 原生指令 `app.use('/images', express.static(path.join(__dirname, 'public/images')))` 將實體照片資料夾對外開放。
2.  **上傳照片 API**：導入 `multer` 工具，實作 `/api/admin/items/upload-image` 端點。當管理員在上傳照片時，API 會自動依照 `Barcode` 將圖片重新命名存檔，再將路徑寫入該料件的 `image_url`。

### 階段三：前端介面與防呆體驗 (Frontend UI/UX)
1.  **主表單顯示**：在「料件主檔」及「庫存查詢」表格中新增縮圖欄位，支援滑鼠懸停 (Hover) 放大的放大鏡體驗。
2.  **出入庫視覺核對 (核心目標)**：
    *   作業員在「單筆出庫 / 入庫」或使用「無硬體手機跨平台掃描」時，一旦條碼掃過，或是手動輸入完畢。
    *   系統不僅帶出料名與庫存，畫面右側或重點區域會立刻顯示出**高畫質的料件原型照片**。
    *   這能達成「肉眼防呆」的效果：如果系統顯示的相片是一顆馬達，作業員手上拿的卻是一條皮帶，能即時發現拿錯料。
3.  **無圖片防呆處理 (Fallback)**：若該料號尚未由系統管理員建檔照片，React 元件會呈現一個有質感的「無圖片 (No Image Available)」佔位符，不會有破圖紅叉的廉價感。

---

## 三、其他管理維度考量

*   **Excel 大量匯入相容性**：若有數千種料件要一次性上圖片，可以定義一種「同名映射規則」。只要把五千張照片全部命名為「對應的料號.jpg」，直接丟進伺服器資料夾，WMS 就能自動比對掛載，免去逐一上傳的苦力活。
*   **備份機制拆分**：因為採用方案 B 解耦設計，未來的 `warehouse.db` 依然可以只備份這幾 MB 的純文字主檔；而肥大的照片資料夾，可讓 IT 單位排程用增量備份 (Incremental Backup) 即可，降低維運難度。
