# 目前專案及各式專案資料庫選用原則及優缺點比較

本文件分析當前專案（智慧倉儲動態管理系統 WMS）以及常見專案套用的資料庫選型原則，並比較各主流資料庫的優缺點。

---

## 1. 智慧倉儲動態管理系統 (WMS) 資料庫選用原則

倉儲系統本身對於**資料一致性 (Data Consistency)**、**交易完整性 (ACID)** 以及**併發處理 (Concurrency)** 有極高要求（例如：避免多個使用者同時扣減同一筆庫存導致超賣或庫存錯誤）。

### 核心原則：
*   **強大關聯性與交易支援 (ACID)**：必須選擇支援完整交易機制的關聯式資料庫 (RDBMS)。
*   **高併發讀寫**：進出庫頻繁，鎖表機制 (Locking) 必須精細（如行級鎖 Row-level locking），避免整個資料表被鎖死。
*   **資料備份與還原**：需有完善的備份機制及 Point-in-time recovery，防止災難性資料遺失。
*   **報表與複雜查詢**：WMS 常需進行進出庫統計、庫存盤點等複雜的 `JOIN` 與聚合查詢，資料庫引擎需具備優秀的查詢最佳化能力。

---

## 2. 各式專案常見資料庫選用指南與比較

以下列出常見資料庫，並依據其特性、優缺點及適用場景進行分析。

### 2.1 SQLite
**定位**：輕量級、無伺服器 (Serverless)、基於單一檔案的關聯式資料庫。

*   **優點**：
    *   零配置，隨插即用，無須安裝獨立伺服器。
    *   極度輕量，適合嵌入式系統或單機應用。
    *   備份簡單（只需複製單一檔案 `.sqlite`）。
*   **缺點**：
    *   不適合高併發寫入（寫入時會鎖定整個資料庫檔案 Database-level lock）。
    *   缺乏使用者權限管理機制與進階管理工具。
    *   不適合存放海量資料。
*   **適用場景**：
    *   本機開發與測試環境（快速建立 mock data）。
    *   手機 APP 本機端儲存 (iOS/Android)。
    *   輕量級桌面應用程式或小型內部工具。
*   **對本專案 (WMS) 的適用性**：**極低**。WMS 需要處理多人同時操作庫存，SQLite 的鎖定機制會造成嚴重效能瓶頸與超時問題。

### 2.2 PostgreSQL
**定位**：功能最強大、開源的關聯式資料庫系統 (ORDBMS)。

*   **優點**：
    *   完全遵守 SQL 標準與 ACID 屬性，資料一致性極佳。
    *   支援進階資料型態（JSON, Array, PostGIS 空間資料等）。
    *   絕佳的高併發處理能力與效能（MVCC 多版本並行控制機制）。
    *   開源且免費，社群龐大且有眾多強大擴充套件。
*   **缺點**：
    *   架構較為龐大，記憶體與資源消耗略高於 MySQL。
    *   設定與維護的學習曲線相對陡峭。
*   **適用場景**：
    *   **企業級應用、金融系統、WMS 倉儲系統。**
    *   需要處理複雜查詢、高度資料完整性與地理空間分析的專案。
*   **對本專案 (WMS) 的適用性**：**極高**。非常適合做為 WMS 的核心資料庫。

### 2.3 MySQL / MariaDB
**定位**：最普及的開源關聯式資料庫系統，主打快速與易用。

*   **優點**：
    *   讀取效能優異，尤其在 Web 應用程式 (LAMP/LEMP stack) 表現極佳。
    *   生態系極其龐大，幾乎所有語言和框架都有最完善的支援。
    *   設定相對簡單，有大量的 GUI 工具（如 phpMyAdmin, DBeaver）。
*   **缺點**：
    *   複雜查詢（如多重 JOIN 或大量聚合）最佳化不如 PostgreSQL。
    *   對標準 SQL 的嚴格程度較為寬鬆（部分設定下可能允許不完全合法的隱式轉換）。
*   **適用場景**：
    *   一般電商網站、內容管理系統 (CMS, 如 WordPress)。
    *   讀多於寫的 Web 應用程式。
*   **對本專案 (WMS) 的適用性**：**高**。若使用 InnoDB 引擎（支援事務與行級鎖），也能充分勝任 WMS 系統。

### 2.4 Microsoft SQL Server (MSSQL)
**定位**：微軟推出的商用級關聯式資料庫。

*   **優點**：
    *   與 .NET / C# 生態系整合度完美，開發極致順暢。
    *   擁有強大的商業智慧 (BI)、報表服務 (SSRS) 及資料庫管理工具 (SSMS)。
    *   企業級的支援與穩定性，效能極佳。
*   **缺點**：
    *   授權費用昂貴（雖然有免費的 Express 版，但有資料庫大小與資源限制）。
    *   強烈綁定 Windows 生態（現雖支援 Linux，但最佳體驗仍在 Windows）。
*   **適用場景**：
    *   以 C# / .NET 為主力的企業內部系統、ERP、WMS。
    *   有足夠預算且依賴微軟技術棧的大型企業環境。
*   **對本專案 (WMS) 的適用性**：**極高**（若後端採用 .NET Core / C#）。

### 2.5 MongoDB
**定位**：主流的 NoSQL 文件型資料庫 (Document Database)。

*   **優點**：
    *   無固定 Schema，資料結構靈活（以 JSON/BSON 格式儲存），適合快速迭代的敏捷開發。
    *   極佳的水平擴展性 (Horizontal Scalability) 與分片 (Sharding) 能力。
    *   寫入效能高，適合儲存日誌或非結構化資料。
*   **缺點**：
    *   不支援傳統的 JOIN（需在應用層處理或使用 `$lookup`，效能較差）。
    *   不以 ACID 強一致性見長（雖然新版已支援多文檔事務，但仍非強項）。
*   **適用場景**：
    *   內容型網站、物聯網 (IoT) 數據收集、大型日誌系統、即時分析系統。
*   **對本專案 (WMS) 的適用性**：**偏低**。庫存管理極度依賴關聯性與事務鎖定，NoSQL 不適合做為 WMS 的核心交易資料庫，但可作為**附屬資料庫**（例如：儲存系統操作日誌、硬體設備 Log 等）。

### 2.6 Redis
**定位**：基於記憶體 (In-Memory) 的鍵值 (Key-Value) 資料庫。

*   **優點**：
    *   極致的讀寫速度（微秒級），因為所有資料皆存放在雷射記憶體。
    *   支援豐富的資料結構（字串, 列表, 集合, Hash 等）。
*   **缺點**：
    *   資料容量受限於實體伺服器的記憶體大小。
    *   資料持久化機制雖然有（RDB, AOF），但主要仍視為快取用途。
*   **適用場景**：
    *   資料快取 (Caching)、Session 管理。
    *   分散式鎖 (Distributed Lock)、訊息佇列 (Message Queue)。
*   **對本專案 (WMS) 的適用性**：**高（作為輔助）**。可用來做 Token 管理、熱點庫存資料快取，或處理高併發下的**分散式鎖**（防止同時扣減庫存）。

---

## 3. 總結與建議

1. **WMS 核心資料庫選擇**：
    *   首選 **PostgreSQL** 或 **MySQL (InnoDB)**。
    *   若團隊以 C#/.NET 為主且企業有授權，**SQL Server** 也是極佳選擇。
2. **絕對避免**：
    *   **SQLite**（無法處理併發寫入）
    *   **MongoDB** 作為「核心庫存結算庫」（難以確保嚴格一致性）
3. **混合架構 (Polyglot Persistence)**：
    *   核心資料、庫存、訂單 ➔ 關聯式資料庫 (PostgreSQL/MySQL)。
    *   高併發快取、Token 與分散式鎖 ➔ Redis。
    *   大量操作日誌與非結構紀錄 ➔ MongoDB（非必要，但可選）。
